# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**UIGen** ŌĆö an AI-powered React component generator. Users describe components in a chat, Claude generates code using a virtual file system, and a live preview renders it inside an iframe using browser-native ES modules.

## Commands

```bash
# First-time setup (installs deps, generates Prisma client, runs migrations)
npm run setup

# Development server (Turbopack, Windows-compatible)
npm run dev

# Build
npm run build

# Lint
npm lint

# Run all tests
npm test

# Run a single test file
npx vitest run src/path/to/file.test.ts

# Reset database
npm run db:reset

# Regenerate Prisma client after schema changes
npx prisma generate
npx prisma migrate dev
```

> On Windows, `npm run dev` uses `set NODE_OPTIONS=...` syntax. The `build`/`start` scripts use Unix syntax ŌĆö run those in Git Bash or WSL if needed.

## Architecture

### Data flow

1. **Chat UI** (`ChatContext`) calls `POST /api/chat` with the conversation `messages` and the serialized virtual file system (`files`).
2. **API route** (`src/app/api/chat/route.ts`) reconstructs a `VirtualFileSystem` from the payload, calls `streamText` (Vercel AI SDK) with two tools exposed to Claude: `str_replace_editor` and `file_manager`.
3. **Claude** streams back tool calls; the server executes them against the in-memory `VirtualFileSystem` and streams results back.
4. **Client** receives tool call events via `onToolCall` in `useAIChat` ŌåÆ forwards them to `FileSystemContext.handleToolCall`, which mirrors the same mutations client-side.
5. **PreviewFrame** watches `refreshTrigger` from `FileSystemContext`, transpiles all files with Babel Standalone, builds a browser import map (with esm.sh for third-party packages), and writes the result into an `<iframe srcdoc>`.

### Key abstractions

| Module | Purpose |
|---|---|
| `src/lib/file-system.ts` | `VirtualFileSystem` ŌĆö in-memory tree, no disk I/O. Canonical source of truth for generated files. |
| `src/lib/transform/jsx-transformer.ts` | Transpiles JSX/TSX with Babel Standalone, resolves imports, builds import map + preview HTML for the iframe. |
| `src/lib/tools/str-replace.ts` | `str_replace_editor` tool definition (view/create/str_replace/insert). |
| `src/lib/tools/file-manager.ts` | `file_manager` tool definition (rename/delete). |
| `src/lib/contexts/file-system-context.tsx` | React context wrapping `VirtualFileSystem`; exposes `handleToolCall` so client mirrors server-side mutations. |
| `src/lib/contexts/chat-context.tsx` | Thin wrapper around `useAIChat`; passes serialized FS with every request; tracks anonymous work. |
| `src/lib/provider.ts` | Returns real `anthropic(MODEL)` when `ANTHROPIC_API_KEY` is set, otherwise `MockLanguageModel`. Mock is useful for offline development. |
| `src/lib/prompts/generation.tsx` | System prompt for Claude: always create `/App.jsx` as entry point, use `@/` alias for local imports, Tailwind for styling. |
| `src/lib/auth.ts` | JWT sessions via `jose`, stored as an httpOnly cookie (`auth-token`). Server-only. |
| `src/middleware.ts` | Protects `/api/projects` and `/api/filesystem` routes. |

### File system rules (important for prompt/tool work)

- Entry point is always `/App.jsx` (or `.tsx`).
- Local imports must use the `@/` alias (e.g., `@/components/Button`).
- No HTML files ŌĆö the iframe boilerplate is generated by `createPreviewHTML`.
- Third-party packages are resolved via `https://esm.sh/<package>` at preview time; they do **not** need to be in `node_modules`.

### Database

SQLite via Prisma. Schema is at `prisma/schema.prisma`; generated client is output to `src/generated/prisma/`. Two models: `User` (email+bcrypt password) and `Project` (stores serialized messages + VFS snapshot as JSON strings).

### AI provider

Model: `claude-haiku-4-5` (defined in `src/lib/provider.ts`). When `ANTHROPIC_API_KEY` is absent, `MockLanguageModel` returns static counter/form/card components ŌĆö useful for development without burning API credits.

## Testing

Tests use Vitest + jsdom + React Testing Library. Test files live alongside source in `__tests__/` subdirectories. The vitest config is `vitest.config.mts`.
